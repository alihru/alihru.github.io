Introduction to rESEF

0 Introduction

In the recent years the European Single Electronic Standard (ESEF) was introduced by the European Securities and Markets Authority (ESMA). According to the mandate, all annual financial reports of issuers within scope must be prepared in EXtensible HyperText Markup Language(XHTML) format. The data within the report must be tagged with eXtensible Business Reporting Language (XBRL) tags, embedded into the XHTML documents using Inline XBRL technology. The tags should be based on the ESEF taxonomy, which is largely based on the International Financial Reporting Standards (IFRS) taxonomy published by the IFRS Foundation but with some additions on behalf of ESMA.

The main goal of rESEF is to automise the process of retrieval of the aforementioned tagged financial data. With the use of rESEF, the reliance on external sources (such as Bloomberg and Refinitiv) and on manual data extraction can be reduced, such that up-to-date, high-quality data can be obtained as soon as the reports are published by the respective issuers.

Furthermore, rESEF contains functions and (S3) methods for computation of certain financial ratios (EBITDA margin, return on equity, price-to-book, price-to-earnings, enterprise value to sales) and provision of data tagged with specific tags.

1 Usage

1.1 Set-up

The package utilises a number of R packages, but only imports selected functions from each of them. It is assumed that the required packages are installed. Otherwise, run the following command before loading in the package:

# set wd appropriately
devtools::install(pkg = "rESEF", 
                  dependencies = c("Imports", "Depends", "Suggests"))

Now, load in the package:

library(rESEF)

1.2 Extracting information from ESEF reports

The main function within the package is load_reports. It extracts information from all HTML and XHTML files (corresponding to annual financial reports in ESEF format) within a given directory.

Two inputs must be supplied:

  1. path: character string; the path to the directory, where XHTML/HTML files, corresponding to ESEF annual financial reports of a sample of companies, are stored.

    • It is recommended that the user ensures that there are no XHTML/HTML files which do not correspond to annual financial reports in ESEF format within the directory.

    • Usually, when ESEF reports are downloaded, all files corresponding to a single report are compressed into a .zip file. The user must uncompress the relevant files before running the function.

    • The XHTML/HTML files can be stored in the directory as either standalone files or else within subdirectories.

  2. year: character or numeric; the starting year of the reporting period of interest.

There are two additional inputs which may be specified but have default values:

  1. save.texts: logical; if TRUE, all visible text within the reports will be extracted and saved into a ‘rSEEF_list_texts’ class object, within the ‘rESEF_list’ class object returned by the function.

    • By default, save.texts = FALSE to save time and memory when loading in the reports.

  2. verbose: logical; if TRUE, descriptive messages about the progress of the loading process are returned.

For the example considered in the vignette we did the following:

  • Download annual financial reports prepared in ESEF format for 2020 in German and/or English for 3 companies, namely Immofinanz AG, Schoeller-Bleckmann Oilfield Equipment AG and Andritz AG.

  • Unzip the downloaded .zip files and put them into the same folder. Let DATA_DIRECTORY_PATH correspond to the path to this folder.

Now, we are ready to use the package. Load in the reports as follows:

# name the object ATX_sample since we are dealing with a sample of ATX companies
ATX_sample = load_reports(path = DATA_DIRECTORY_PATH, 
                          year = 2020, 
                          save.texts = TRUE, 
                          verbose = FALSE)

Note that user’s interaction may be required at the final stage of the loading process. The user may be asked to examine parts of tibbles, containing numeric values for some companies. This happens when an unknown or no format is provided for some values. The user is asked to examine a tibble of some or all of such values and type ‘1’, if “comma formatting” should be applied (i.e. if the user believes that comma is used as a decimal operator), or ‘2’, if “dot formatting” should be applied (i.e. if the user believes that dot is used as a decimal operator). Both values are to be provided as numeric. More details on formatting to be provided in Section 2 (see section 4.1 of https://www.xbrl.org/guidance/ixbrl-tagging-features/ to understand what ‘format’ means in our context).

1.3 Filter for specific values

We can use the function get_value to filter facts tibbles for values tagged with specific tags. The function takes the following inputs:

  1. object: object of class ‘rESEF_list’ or ‘rESEF_list_facts’; error occurs if ‘rESEF_list’ does not contain a ‘rESEF_list_facts’ object.
  2. company: character or numeric; the company’s name, as returned by get_company_names, or the position of the company in the ‘rESEF_list_facts’ object corresponding to the supplied object, respectively.
  3. tag: character vector; the tag(s) of interest.
  4. year: character or numeric; year for the period or the single date to which the value(s) should correspond.

The user must supply the tags together with their corresponding namespaces e.g. “ifrs-full:Equity”. The tibbles returned are subsets of the facts tibbles stored within the corresponding ‘rESEF_list_facts’ objects.

It is important to note, that the function also filters for period as follows:

  • if the value corresponds to a period, it requires the start date to be in January of the requested year and the end date to be in December of the requested year,

  • if the value corresponds to a single date, it requires the date to be in December of the requested year.

If no filtering for period is required, the user is encouraged to make use of the functionalities of the dplyr package to filter the tibbles his/her own way.

Revenue2019 = get_value(object = ATX_sample,

1.4 Compute financial ratios

Lastly, we can compute a number of financial ratios, based on the values extracted from the reports. For this, one should make use of the function get_ratio, if a ratio value for one company is required, and get_ratios, if some ratio values are needed for several companies.

The two functions share most of the inputs:

  1. object: object of class ‘rESEF_list’ or ‘rESEF_list_facts’; error occurs if ‘rESEF_list’ does not contain a ‘rESEF_list_facts’ object.

  2. ratio: character; ratio to be computed; takes on one of the following values:

    • “EBITDA_margin” (EBITDA margin),

    • “ROE” (return on equity),

    • “PE” (price-to-earnings),

    • “PB” (price-to-book),

    • “EVSales” (enterprise value to sales).

  3. year: character or numeric; the ratio is calculated for the end of this year.

  4. date: character; only needed in the case of the share price dependent ratios (“PE”, “PB”, “EVSales”), ignored for other ratios; to be supplied in the format “YYYY-MM-DD”; set to as.character(Sys.Date()) by default.

For the get_ratio function, one needs to also supply a value for company. We refer the reader to Section 1.3 for details.

What form does the output take? A tibble with the following columns is returned:

  • “company” - company name

  • “year” - year requested

  • “ratio” - ratio requested

  • “value” - value of ratio computed

In the case of the share price dependent ratios (“PE”, “PB”, “EVSales”), there are 2 extra columns:

  • “date_requested” - date, closing price of which was requested to be used for computation

  • “date_used” - date, closing price of which was used for computation

The need for the last pair is justified by closing prices not always being available for particular dates e.g. if the requested date is too close to the present day.

The following details should be noted:

  • NA values are returned in the “value” and “date_used” column if relevant information could not be extracted from the report.

  • The user can only request one kind of ratio to be computed at a time.

In the course of our analysis of the 2020 ATX reports, we observed that NA values are unfortunately frequent. ESMA states that where “the issuer prepares IFRS consolidated financial statements, it shall mark up these IFRS consolidated financial statements using the XBRL markup language”, but nevertheless many numerical values:

  • are left untagged (for example, in the case of numbers of shares outstanding), since many important values occur within the text of the report or else within “secondary” statements, or

  • are tagged with an own tag; such values are ignored for ratio computations, as further analysis of the tag description by a human reader would be required.

2 Data pipeline

Now that we have shown the possible uses of rESEF, we expand on the thought process behind the construction of the so-called data pipeline, i.e. we answer the question of how one can go from the zipped files constituting the issuers’ ESEF report and taxonomy files to computation of ratios based on the information extracted from these files.

Loading in reports

The function load_reports takes the given path and considers each file/directory within the corresponding directory, one at a time. If there is one or more HTML or XHTML file(s) corresponding to a file/directory, their paths are taken one at a time and the corresponding files are read in as xml files and stored as objects of class ‘xml_document’.

The node with the name “NameOfReportingEntityOrOtherMeansOfIdentification” (a mandatory tag) is found within the ‘xml_document’ object. It contains the name of the company, as well as the value of the “contextRef” attribute. This value is the value of the “id” attribute of the node (whose node name is “context”), which contains information on the reporting period (start and end dates). If the year of the start and end dates is the same as the requested year, we do further work with the ‘xml_document’ object. Otherwise, we throw it out and move on to the next file/directory within the provided directory.

For each of the files with correct reporting period, the following is done.

  1. Extract all information

    First, we discuss how all the needed information is extracted.

  • All the nodes within the ‘xml_document’ object are found and stored, as well as the names of all the unique attributes, whose values are used to provide description for these nodes.

  • A large tibble is created, which stores information for each node within the report. The column names are the unique attribute names within the report. If some attribute’s value is not specified for the node, an NA appears in the respective row (node) and column (attribute). Additionally, there is a column called “text”. All of the text nested within a node is saved in the corresponding cell. These text values are only used internally.

  • A subset of this “report tibble” is selected by filtering for all nodes with the node name “nonFraction”. All tagged numerical values are stored as text within these nodes. Additionally, the attributes and their values describe what tag was used, how the value is formatted and scaled, what the sign of the value is, etc. We call these subtibbles “facts tibbles”. They are stored as separate objects from now on.

  • The “nonFraction” nodes often have “contextRef” and “unitRef” values. These point to other nodes, with node names “context” and “unit”, respectively. Within these nodes, the information on the periods and currencies (eg. EUR vs EUR per share) corresponding to the values is stored. For the “context nodes”, a list of tibbles is created, with a tibble for each node. These will later be used to produce the final tibble, returned by the function.

  • If the user wishes to save texts as well, the node with the name “body” is found. There are usually many nodes which are children of this node. The nodes whose text is not displayed are ignored. The texts for all other children are collapsed appropriately into a single character string. These character strings will be returned by the function.

  1. Append or not?

    Suppose there are several XHTML files for the same company and in the same language. We want to append the corresponding objects (whose creation was described above) only once. How can we do that?

  • Apart from the period information, the “context” nodes also store the LEI identifier value for each company.

  • We check if the identifier-language combination has already occurred. If it did, then the objects are thrown out and we move on to the next file. Otherwise, the 4 objects (“report tibble”, “facts tibble”, list of “context tibbles”, report text string) are attached to 4 separate lists.

    Once the above steps have been completed for all (X)HTML files within the supplied directory, a new list is created which holds the 4 lists.

  1. Putting “facts tibbles” into their final form

    Now that we have the information for all of the companies, we need to “connect the dots”. We alter the “facts tibbles” by using the additional information related to the “nonFraction” nodes which is stored in the other objects (“report tibbles” and lists of “context” tibbles).

  • The LEI number of the company is stored within each “context” node and,hence, easily extracted.

  • In order to connect the company to a ticker, the LEI number is matched to a ticker through the CONST$COMPANY_IDS data frame (already discussed in Section 1), which needs to be maintained by the user and originally contains correspondences for the ATX companies only.

  • The periods are filled out by matching the “nonFraction” nodes to their “context” nodes through “contextRef”. The “context” nodes sometimes store further description of the nature of the value (e.g. whether a value tagged as equity refers to issued capital, retained earnings etc.). This information is also extracted and used to fill the “explicitMember_dimension” and “explicitMember_text” columns (discussed in Section 1).

  • The currencies are filled out by matching the “nonFraction” nodes to their “unit” nodes through “unitRef”.

  • Appropriate formatting methods are applied to the numerical values, as specified by the “format” attribute. As the transformation registry for XBRL tagging is regularly updated, the user needs to update the formatting methods in use by changing CONST data appropriately.

  • The values are scaled (according to the “scale” attribute values) and appropriate signs are applied.

  • Lastly, columns which are no longer needed (e.g. “format”, “column”, “scale”) are removed.

  1. New classes

    The altered “facts tibbles” are nested within a single list, the class of which is defined to be ‘rESEF_list_facts’. The text character strings are nested within another list of class ‘rESEF_list_texts’. A new list is created to hold the 2 lists and assigned the class ‘rESEF_list’. “Report tibbles” and lists of “context” tibbles are thrown out as we no longer need them.

Values and ratios

The function get_value simply filters the tibbles for the right name/names (tag/tags), as well as the period (discussed in Section 1.3).

The workings of get_ratio and get_ratios functions are less straight-forward.

  • An inner function, analogous to get_value is defined to retrieve end-of-period values or values corresponding to the whole January - December period.

  • For each ratio there are several values which are used for computation. For each of such values, a more complex variant of the inner function discussed above is defined, which attempts to use the formulas connecting the relevant IFRS taxonomy elements in order to retrieve the needed value.

    For example, suppose we need the value tagged with “ifrs-full:OtherFinancialLiabilities”, but this tag is not used. Instead, we can look for values tagged with “ifrs-full:OtherCurrentFinancialLiabilities” and “ifrs-full:OtherNoncurrentFinancialLiabilities” and sum the corresponding values up if they are present. All such formulas are defined in IFRS formula linkbases (found within IFRS taxonomy file collections). One needs to note that different taxonomies may use different labels and formulas. As these may change in the future, the functions would need to be adjusted accordingly.

Report issuers are also allowed to use their own tags if the value they want to tag does not match any of the core IFRS taxonomy elements. Unfortunately, we have to ignore these values, as by the current ESMA regulations require these to be either wider or narrower in the accounting meaning and scope than the IFRS taxonomy elements.

For the share price dependent ratios, the prices are fetched from the API of Yahoo Finance.

3.1 Financial ratios

Suppose we ask ourselves the question: what is the value of automising the process? For example, assume that 5 out of 20 ATX companies publish in May, while the other 15 publish their reports in July. Consider the following two scenarios:

  • We update the values and financial ratios for the 5 companies in May and keep the old ones for the other 15 companies. We then proceed to invest into the “best” subset of companies in May.

  • We wait until July when all reports are in and update all the values and ratios at the same time. We then invest into the “best” subset of companies in July.

Obviously, with the scarcity of values obtained the task becomes chaotic. Our hope was that we would be able to compute all of the financial ratios considered for the 20 ATX companies, with all required values obtained directly from the reports. Although the value is limited, we still wanted to see whether a profitable financial decision could be made.

First, we identify combinations of ratios which can be computed for the largest number of our companies, both for 2019 and 2020 (as the non-updated values should be based on the 2019 reports). It turns out that the ROE (return-on-equity) and P/E (price-to-earnings) combination is the most “popular”, if we assume that ROE values for 2019 are readily available, since the values needed for 2019 ROE computations are generally not found in the 2020 reports. Assuming this, 2019 and 2020 ROE and P/E values are available for a subset of 8 companies. From the table we can see that by July, 19 out of 20 reports are in. Therefore, we adopt the following strategy:

  • Suppose we have the following “report-review” dates: 1st April, 1st May, 1st June and 1st July (2021).

  • For each of these dates, we make 2 tibbles:

    • one, where we update some values if the publishing date was earlier than the “report-review” date, and

    • another, where for all ratios we use the values from last year, i.e. we do not sit down to review reports until July, when (almost) all of them are in.

  • Each company within each tibble receives a “rank” for each of the ratios. We take the simplistic view that the higher the ROE and the lower the P/E is, the better the performance of the company. The “ROE_rank” of the company with the highest ROE would then be 8 and the “ROE_rank” of the company with the lowest ROE would be 1. Analogous ranking is applied for P/E.

  • The ranks are then summed to give the “total rank” and one half of the companies with highest total ranks are invested into, a) in proportion to the values of the total rank and b) in equal proportions.

  • This is done for both updated and not updated tibbles and scatterplots are used to show the performance of the two portfolios over time.

fill_in_values = function(tb, update, ratio){
  
  company_seq = tb[["company"]]
  date_seq = tb[["date"]]
  publishing_date_seq = tb[["publishing_date"]]
    
  if (update){
    year_seq = ifelse(date_seq < publishing_date_seq, 2019, 2020)
  }else{
    # 4 report review dates, 8 companies
    year_seq = rep(2019, 4 * 8)
  }
  
  tb[ratio] = 
    unlist(
      sapply(1:(4*8), simplify = FALSE, 
             function(i){
               if (ratio == "ROE" & year_seq[i] == 2019){
                 rep(ROE2019, 4)[i]
               }else{
                 tmp = get_ratio(object = ATX_subset_facts,
                                 company = company_seq[i],
                                 ratio = ratio,
                                 year = year_seq[i],
                                 date = date_seq[i])
                 tmp[["value"]]
               }
             }
             )
      )
  tb
}

# helper functions

# - create ranks as described earlier
create_ranks = function(tb, n){

  tb = tb %>% arrange(desc(ROE))
  tb["ROE_rank"] = n : 1
  
  tb = tb %>% arrange(PE)
  tb["PE_rank"] = n : 1
  
  tb["rank"] = tb["ROE_rank"] + tb["PE_rank"]
  tb = tb %>% arrange(desc(rank))
  tb
}

# - fetch prices from yahoo
prices = function(date, tickers){
  
  date_lower = date - 10
  date_upper = date + 1
  
  suppressWarnings(
    unlist(
      sapply(tickers, simplify = FALSE,
             function(ticker){
               prices_ = tq_get(ticker,
                                from = as.character(date_lower),
                                to = as.character(date_upper),
                                get = "stock.prices")
               prices_[[nrow(prices_), "close"]]
           }
           )
      )
    )
}




# - compute portfolio value, given date, company tickers and weights (numbers of shares) 
portfolio_value = function(weights, tickers, date){

  return(sum(prices(date, tickers) * weights))
}

# main function
portfolio_values = function(tb_upd, tb_not_upd, date, amount, n){
  
  date = as.Date(date)
  
  tb_upd = tb_upd[tb_upd[["date"]] == date, ]
  tb_not_upd = tb_not_upd[tb_not_upd[["date"]] == date, ]
  
  tb_upd = create_ranks(tb_upd, n)
  tb_not_upd = create_ranks(tb_not_upd, n)
  
  tickers_upd = pull(tb_upd[1:floor(n/2), "ticker"]) 
  tickers_not_upd = pull(tb_not_upd[1:floor(n/2), "ticker"]) 
  
  prices_upd = prices(date, tickers_upd)
  prices_not_upd = prices(date, tickers_not_upd)
  
  
  weights_upd = sapply(1:floor(n/2), function(i){
    (tb_upd[["rank"]][i] / sum(tb_upd[["rank"]][1:floor(n/2)]) * amount) /
      prices_upd[i]})
  weights_not_upd = sapply(1:floor(n/2), function(i){
    (tb_not_upd[["rank"]][i] / sum(tb_not_upd[["rank"]][1:floor(n/2)]) * amount) / 
      prices_not_upd[i]})
  
  # first dates corresponding to next 6 months
  dates = purrr::reduce(lapply(0:6, function(i) date %m+% period(i, "month")), c)
  
  tibble(
    date = dates,
    updated_portfolio_value = 
      sapply(dates, function(date) portfolio_value(weights_upd, tickers_upd, date)),
    not_updated_portfolio_value = 
      sapply(dates, function(date) portfolio_value(weights_not_upd, tickers_not_upd, date))
  )
}

3.2 ESG

In this part basic sentiment analysis (SA) methods are used to compute the relative frequencies of ESG related words within the reports of ATX companies. For the analysis of the reports written in English, all terms (482 in total) of the ESG word list, put together by P. Baier et. al (), are used. To conduct SA on the German reports, this list was translated with the use of the DeepL translator (https://www.deepl.com/translator).

Some small adjustments were made to the translated list to account for gendering in the German language, for example, the words “inspector” and “inspectors” were present in the English word list, were translated to “Inspektor” and “Inspektoren” by the DeepL Translator and we added “Inspektorin” and “Inspektorinnen” to the translated list. Additionally, we examine the effect of stemming on the German reports and the translated word list to account for declension in the German language.

DE_tf_vecs = lapply(DE_tf_vecs, function(tf_vec) tf_vec[!(tf_vec == 1 | tf_vec == 2)])

# remove German stopwords
DE_tf_vecs = lapply(DE_tf_vecs, function(tf_vec){
  tf_vec[!(names(tf_vec) %in% stopwords(language = "de"))]
})

# create list of vectors of stemmed terms
DE_tf_vecs_stem = lapply(DE_tf_vecs, function(tf_vec){
  names(tf_vec) = stemDocument(names(tf_vec), lang = "de")
  tf_vec
})


# compute relative frequencies of the ESG words for the German texts 
ESG_DE = 
  tibble(company = names(DE_tf_vecs), 
         # ESG_perc with no stemming
         ESG_perc = sapply(DE_tf_vecs, function(tf_vec){
           sum(tf_vec[which(names(tf_vec) %in% ESG_words_DE)]) / sum(tf_vec)}),
         # ESG_perc with stemming
         ESG_perc_stem = sapply(DE_tf_vecs_stem, function(tf_vec){
           sum(tf_vec[which(names(tf_vec) %in% ESG_stems_DE)]) / sum(tf_vec)}))

In both cases, the majority of the companies, the stock of which was labelled as an ESG compliant investment by RBI, are found in the top half of the table. The “green-leaf” companies on average have higher shares of ESG words on total words in their reports.

In addition to the limitations associated with the first part of the experiment on English reports (2020 reports vs. 2022 recommendation list, small sample size), one has to question the validity of construction of a word list by means of translating a word list dedicated to the same purpose but in a different language. Additionally, stemming might mean that occurrences of words not connected to company’s ESG-related statements and actions count towards the ESG “score”.

Putting these limitations aside, the results observed can indicate that, again, companies with a stronger (relative to other companies) focus on ESG-related topics are more likely to capture this in their annual financial reports through the use of appropriate vocabulary.
